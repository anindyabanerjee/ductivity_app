Section 01

Introduction to React Native

What is React Native?

React Native lets you build mobile apps for iOS and Android using JavaScript and React. Unlike a web-wrapped app, React Native renders real native components -- a <View> becomes a native UIView on iOS and an android.view.View on Android.

React Native vs React Web

  • No HTML elements -- you use View instead of div, Text instead of p or span
  • StyleSheet API -- CSS-like objects, but no cascading or class selectors
  • No browser APIs -- no DOM, no window, no document
  • Platform-specific code -- some APIs differ between iOS and Android

Expo vs Bare React Native

Expo provides a managed workflow with pre-configured tools, OTA updates, and a development client (Expo Go) for quick testing. Ductivity uses Expo, which means we can test on a physical device by simply scanning a QR code.

Expo Go Limitations

Expo Go does not support native modules like react-native-reanimated or push notifications. Ductivity uses the built-in Animated API instead of Reanimated for exactly this reason. Push notifications require a development build.

React Native uses native components under the hood, not a WebView. Expo provides a streamlined development experience, but some native features require a custom development build rather than Expo Go.

Core Components

React Native provides a set of core components that map to native UI widgets. Here are the most important ones, all used in Ductivity.

View and Text

View is the fundamental building block -- equivalent to div in web. Text is required for any text content (you cannot put raw text inside a View).

TaskScreen.tsx -- Header 
<View style={styles.container}>
  <StatusBar barStyle="light-content" />
  <Text style={styles.header}>
    {userName ? `Hey ${userName}, what are you doing?` : 'What are you doing?'}
  </Text>
  <Text style={styles.subheader}>Tap to log your current activity</Text>
</View>

TouchableOpacity

The standard tappable component. It reduces the opacity when pressed, providing visual feedback. Used throughout Ductivity for activity cards.

TaskScreen.tsx -- ActivityCard 
<TouchableOpacity
  style={[
    styles.activityCard,
    { borderLeftColor: CATEGORY_COLORS[activity.category] },
    isActive ? styles.activeCard : null,
  ]}
  onPress={onPress}
  activeOpacity={0.7}
  disabled={isLoading}
>
  <Text style={styles.activityEmoji}>{activity.emoji}</Text>
  <Text style={styles.activityName}>{activity.name}</Text>
</TouchableOpacity>

FlatList -- Efficient List Rendering

FlatList only renders visible items, making it performant for long lists. Ductivity uses it with numColumns={2} for the activity card grid.

TaskScreen.tsx -- FlatList Grid 
<FlatList
  data={ACTIVITIES}
  renderItem={renderCard}
  keyExtractor={(item) => item.id}
  numColumns={2}
  columnWrapperStyle={styles.row}
  scrollEnabled={false}
  removeClippedSubviews={false}
  ListFooterComponent={
    lastLogged ? (
      <Text style={styles.lastLoggedText}>Last logged: {lastLogged}</Text>
    ) : null
  }
/>

ScrollView with Pull-to-Refresh

For content that needs vertical scrolling, ScrollView is the go-to. The Dashboard uses it with a RefreshControl for pull-to-refresh.

DashboardScreen.tsx -- ScrollView 
<ScrollView
  style={styles.chartsContainer}
  showsVerticalScrollIndicator={false}
  refreshControl={
    <RefreshControl
      refreshing={refreshing}
      onRefresh={onRefresh}
      tintColor="#e94560"
    />
  }
>
  {loading ? <SkeletonLoader /> : <ActivityChart logs={logs} />}
</ScrollView>
Key Takeaway

Use FlatList for dynamic lists (it virtualizes rows for performance). Use ScrollView for static content that just needs to scroll. Never nest a FlatList inside a ScrollView with the same scroll direction.

Styling in React Native

StyleSheet.create

React Native styles use JavaScript objects, not CSS files. StyleSheet.create validates styles at creation time and enables optimizations.

TaskScreen.tsx -- Styles 
const styles = StyleSheet.create({
  container: {
    flex: 1,
    backgroundColor: '#1a1a2e',
    padding: CARD_PADDING,
    paddingTop: 60,
  },
  activityCard: {
    width: CARD_WIDTH,
    backgroundColor: '#16213e',
    borderRadius: 12,
    padding: 16,
    borderLeftWidth: 4,
    alignItems: 'center',
  },
});

Dynamic Card Width with Dimensions

To build a responsive two-column grid, Ductivity calculates card widths based on the screen dimensions.

TaskScreen.tsx -- Dynamic Width 
const SCREEN_WIDTH = Dimensions.get('window').width;
const CARD_GAP = 12;
const CARD_PADDING = 20;
const CARD_WIDTH = (SCREEN_WIDTH - CARD_PADDING * 2 - CARD_GAP) / 2;

This formula subtracts horizontal padding and gap from the total width, then divides by 2 for two columns. The result adapts to any screen size.

Conditional Styles

React Native supports style arrays. Pass null or undefined to conditionally include a style:

TaskScreen.tsx -- Conditional Styling 
style={[
  styles.activityCard,
  { borderLeftColor: CATEGORY_COLORS[activity.category] },
  isActive ? styles.activeCard : null,
]}

Dark Theme Pattern

Ductivity uses a consistent color palette across all screens: #1a1a2e for backgrounds, #16213e for cards, #e94560 as the accent, and #a0a0b0 for secondary text. This is applied via StyleSheet.create at the bottom of each screen file.

Key Takeaway

Flexbox is the only layout engine in React Native (it defaults to flexDirection: 'column'). Use Dimensions.get('window') for responsive sizing. Style arrays let you compose base styles with conditional overrides cleanly.

useState for Local State

The TaskScreen manages several pieces of local state. Each useState call handles one concern:

const [lastLogged, setLastLogged] = useState<string | null>(null);

const [cooldownEnd, setCooldownEnd] = useState<number>(0);

useEffect for Side Effects

Ductivity uses multiple useEffect hooks to manage timers, data fetching, and animations. The cooldown timer is a great example:

TaskScreen.tsx -- Cooldown Timer 
// Cooldown countdown -- pauses when screen loses focus
useEffect(() => {
  if (!isFocused || cooldownEnd <= Date.now()) return;

  const interval = setInterval(() => {
    const remaining = cooldownEnd - Date.now();
    if (remaining <= 0) {
      setCooldownText('');
      clearInterval(interval);
    } else {
      setCooldownText(formatCountdown(remaining));
    }
  }, COOLDOWN_TICK_INTERVAL);

  return () => clearInterval(interval);
}, [cooldownEnd, isFocused]);

useRef for Persistent Values

useRef stores values that persist across renders without causing re-renders. In Ductivity, it holds Animated values and a flag to prevent re-running entrance animations:

DashboardScreen.tsx -- useRef 
const hasAnimated = useRef(false);

useEffect(() => {
  if (!hasAnimated.current) {
    hasAnimated.current = true;
    // Run entrance animations only once
    header.animate();
    subheader.animate();
  }
}, []);

Context API for Global State

Ductivity uses React Context to share the username across all screens without prop drilling:

context/UserContext.tsx 
interface UserContextType {
  userName: string;
  setUserName: (name: string) => Promise<void>;
  isLoading: boolean;
}

export function UserProvider({ children }: { children: React.ReactNode }) {
  const [userName, setName] = useState('');
  const [isLoading, setIsLoading] = useState(true);

  useEffect(() => {
    AsyncStorage.getItem(USER_NAME_KEY).then((name) => {
      if (name) setName(name);
      setIsLoading(false);
    });
  }, []);

  const setUserName = async (name: string) => {
    await AsyncStorage.setItem(USER_NAME_KEY, name);
    setName(name);
  };

  return (
    <UserContext.Provider value={{ userName, setUserName, isLoading }}>
      {children}
    </UserContext.Provider>
  );
}

// Consuming in any screen:
const { userName } = useUser();
Key Takeaway

useState for component-level state. useEffect with cleanup functions for timers and subscriptions. useRef for mutable values that should not trigger re-renders. Context for data that many components need (like user identity), paired with AsyncStorage for persistence.

Working with Firebase

Firestore CRUD Operations

Ductivity uses Firestore for persisting activity logs. The service layer abstracts all database operations behind clean async functions.

Creating a Document

services/activityService.ts -- logActivity 
export async function logActivity(
  activityName: string,
  category: CategoryType
): Promise<string> {
  clearCache(); // Invalidate cached dashboard queries
  const docRef = await addDoc(collection(db, COLLECTION_NAME), {
    userId: USER_ID,
    activity: activityName,
    category,
    timestamp: Timestamp.now(),
  });
  return docRef.id;
}

Deleting a Document (Undo)

services/activityService.ts -- deleteActivity 
export async function deleteActivity(docId: string): Promise<void> {
  clearCache();
  await deleteDoc(doc(db, COLLECTION_NAME, docId));
}

Querying with Time Filters

services/activityService.ts -- getActivities 
export async function getActivities(filter: TimeFilter): Promise<ActivityLog[]> {
  const cacheKey = `activities_${filter}`;
  const cached = getCached<ActivityLog[]>(cacheKey);
  if (cached) return cached;

  const startDate = getFilterStartDate(filter);
  const q = query(
    collection(db, COLLECTION_NAME),
    where('userId', '==', USER_ID),
    where('timestamp', '>=', Timestamp.fromDate(startDate)),
    orderBy('timestamp', 'desc')
  );

  const snapshot = await getDocs(q);
  const results = snapshot.docs.map((doc) => ({
    id: doc.id,
    activity: doc.data().activity,
    category: doc.data().category,
    timestamp: doc.data().timestamp.toDate(),
  }));

  setCache(cacheKey, results);
  return results;
}

Data Model

types/index.ts -- ActivityLog Interface 
export interface ActivityLog {
  id?: string;        // Firestore document ID
  userId: string;
  activity: string;
  category: CategoryType;
  timestamp: Date;
}
Key Takeaway

Keep database calls in a service layer (not in components). Always invalidate caches after writes. Use Firestore compound queries with where() and orderBy() for filtered reads. Return document IDs from create operations to enable undo.

Section 07

Animations

Why Built-in Animated API?

React Native has two animation libraries: the built-in Animated API and react-native-reanimated. Ductivity uses the built-in one because Reanimated requires a native module that is not available in Expo Go.

Expo Go Constraint

If you test with Expo Go, stick to the built-in Animated API. If you use a dev build (EAS), Reanimated is available and offers 60fps animations on the UI thread.

AnimatedButton -- Spring Scale

This reusable component gives every button a satisfying press animation with optional haptic feedback:

utils/animations.tsx -- AnimatedButton 
export function AnimatedButton({
  children, style, scaleValue = 0.95, enableHaptic = true, onPress, ...rest
}: AnimatedButtonProps) {
  const scale = useRef(new Animated.Value(1)).current;

  const onPressIn = () => {
    Animated.spring(scale, {
      toValue: scaleValue,
      useNativeDriver: true,
    }).start();
    if (enableHaptic) hapticLight();
  };

  const onPressOut = () => {
    Animated.spring(scale, {
      toValue: 1,
      useNativeDriver: true,
    }).start();
  };

  return (
    <Pressable onPressIn={onPressIn} onPressOut={onPressOut}>
      <Animated.View style={[{ transform: [{ scale }] }, style]}>
        {children}
      </Animated.View>
    </Pressable>
  );
}

useFadeInUp -- Reusable Entrance Hook

utils/animations.tsx -- useFadeInUp 
export function useFadeInUp(delay: number = 0) {
  const opacity = useRef(new Animated.Value(0)).current;
  const translateY = useRef(new Animated.Value(20)).current;

  const animate = () => {
    Animated.parallel([
      Animated.timing(opacity, {
        toValue: 1, duration: 500, delay,
        easing: Easing.out(Easing.cubic),
        useNativeDriver: true,
      }),
      Animated.timing(translateY, {
        toValue: 0, duration: 500, delay,
        easing: Easing.out(Easing.cubic),
        useNativeDriver: true,
      }),
    ]).start();
  };

  return { style: { opacity, transform: [{ translateY }] }, animate };
}

Toast Slide-In Animation

The Toast component combines spring and timing animations for a polished entrance:

components/Toast.tsx -- Animation 
// Slide in with spring + fade
Animated.parallel([
  Animated.spring(translateY, {
    toValue: 0, damping: 14, stiffness: 120,
    useNativeDriver: true,
  }),
  Animated.timing(opacity, {
    toValue: 1, duration: 300,
    useNativeDriver: true,
  }),
  Animated.spring(scale, {
    toValue: 1, damping: 12, stiffness: 100,
    useNativeDriver: true,
  }),
]).start();

// Auto-hide after 4 seconds
timerRef.current = setTimeout(() => hideToast(), 4000);
Key Takeaway

Always set useNativeDriver: true for transform and opacity animations -- it runs them on the native thread for smooth 60fps. Use Animated.spring for natural-feeling interactions and Animated.timing for precise control. Animate once on mount, never on every re-render or focus event.

Section 08

Push Notifications

Expo Notifications Setup

Ductivity uses expo-notifications for local push notifications that remind users to log their activities. The service gracefully degrades when running in Expo Go.

services/notificationService.ts -- Registration 
export async function registerForPushNotifications(): Promise<boolean> {
  if (isExpoGo) {
    console.log('Push notifications not available in Expo Go.');
    return false;
  }
  if (!Device.isDevice) {
    console.log('Push notifications require a physical device.');
    return false;
  }

  const { status } = await Notifications.getPermissionsAsync();
  let finalStatus = status;

  if (status !== 'granted') {
    const { status: newStatus } = await Notifications.requestPermissionsAsync();
    finalStatus = newStatus;
  }

  // Android requires an explicit notification channel
  if (Platform.OS === 'android') {
    await Notifications.setNotificationChannelAsync('activity-reminder', {
      name: 'Activity Reminder',
      importance: Notifications.AndroidImportance.HIGH,
    });
  }
  return finalStatus === 'granted';
}

Scheduling Local Notifications

services/notificationService.ts -- Scheduling 
export async function scheduleActivityReminder(): Promise<void> {
  if (isExpoGo) return;

  await Notifications.cancelAllScheduledNotificationsAsync();

  const settings = await getSettings();
  const intervalMs = getFrequencySeconds(settings.notificationFrequency) * 1000;
  const totalMs = NOTIFICATION_WINDOW_HOURS * 60 * 60 * 1000;

  for (let offset = intervalMs; offset <= totalMs; offset += intervalMs) {
    const triggerDate = new Date(Date.now() + offset);
    if (isTimeBlocked(triggerDate, settings)) continue;

    await Notifications.scheduleNotificationAsync({
      content: {
        title: 'What are you up to?',
        body: 'Tap to log your current activity!',
      },
      trigger: {
        type: Notifications.SchedulableTriggerInputTypes.DATE,
        date: triggerDate,
      },
    });
  }
}

Handling Foreground Notifications

App.tsx -- Notification Listeners 
// Foreground: notification received while app is open
Notifications.addNotificationReceivedListener(() => {
  navigationRef.current?.navigate('Task');
  setNotificationTrigger((prev) => prev + 1);
});

// Background: user tapped a notification
Notifications.addNotificationResponseReceivedListener(() => {
  navigationRef.current?.navigate('Task');
  setNotificationTrigger((prev) => prev + 1);
});
Expo Go Limitation

Push notifications do not work in Expo Go. Guard all notification code with if (isExpoGo) return; to prevent crashes. You need a custom development build via eas build to test notifications.

Key Takeaway

Always check permissions before scheduling. Cancel all existing notifications before re-scheduling to prevent duplicates. Android requires explicit notification channels. Use a context or ref-based counter to communicate notification events across screens.

React.memo

Wrap components in React.memo to skip re-renders when props have not changed. Ductivity memoizes the Toast and ActivityCard:

  activity, isActive, isLoading, onPress,

  activity: typeof ACTIVITIES[0];

  isLoading: boolean;
  onPress: () => void;
}) => (
  <TouchableOpacity ...>

  </TouchableOpacity>

useMemo for Expensive Calculations

The ActivityChart uses useMemo to avoid recomputing chart data on every render:

components/ActivityChart.tsx -- useMemo 
const {
  categoryCounts, totalLogs,
  productivePercent, sortedActivities,
} = useMemo(() => {
  const catCounts = logs.reduce((acc, log) => {
    acc[log.category] = (acc[log.category] || 0) + 1;
    return acc;
  }, {} as Record<CategoryType, number>);

  const total = logs.length;
  const prodCount = logs.filter(l => l.category === 'productive').length;
  const prodPercent = total > 0
    ? Math.round((prodCount / total) * 100) : 0;

  return { categoryCounts: catCounts, totalLogs: total, productivePercent: prodPercent, ... };
}, [logs]);

Query Caching

Ductivity implements a simple TTL-based cache to avoid redundant Firestore reads when users rapidly switch filters:

services/queryCache.ts 
const CACHE_TTL_MS = 30_000; // 30 seconds

const cache = new Map<string, CacheEntry<any>>();

export function getCached<T>(key: string): T | null {
  const entry = cache.get(key);
  if (!entry) return null;
  if (Date.now() - entry.timestamp > CACHE_TTL_MS) {
    cache.delete(key);
    return null;
  }
  return entry.data;
}

export function clearCache(): void {
  cache.clear(); // Invalidate after writes
}
Key Takeaway

Wrap pure components in React.memo. Use useMemo for derived data that is expensive to compute. Cache network responses with a TTL. Use FlatList (not ScrollView + map) for lists, and always provide a stable keyExtractor.

Building & Deploying

EAS Build Configuration

Expo Application Services (EAS) handles building production APKs and AABs. Configure it in eas.json:

eas.json -- Build Profiles 
{
  "build": {
    "preview": {
      "android": {
        "buildType": "apk"
      }
    },
    "production": {
      "android": {
        "buildType": "app-bundle"
      }
    }
  }
}

APK vs AAB

  • APK (Android Package) -- Direct install on devices, good for testing and sideloading
  • AAB (Android App Bundle) -- Required for Google Play Store submission, Google generates optimized APKs per device

Common Build Commands

Terminal 
# Install EAS CLI
npm install -g eas-cli

# Build a preview APK for testing
eas build --profile preview --platform android

# Build for production (AAB for Play Store)
eas build --profile production --platform android

# Run on local device via Expo Dev Client
npx expo start --dev-client
Common Build Issues & Fixes ▼

SDK version mismatch

Ensure your expo version in package.json matches the SDK version in app.json. Run npx expo install --fix to auto-resolve.

Native module not found

If you added a library that needs native linking (like react-native-reanimated), you need a development build -- it will not work in Expo Go.

Gradle build failure

Clear the cache with eas build --clear-cache. Check that your Java version matches what the build expects (Java 17 for newer Expo SDKs).

Key Takeaway

Use EAS Build for both development and production builds. APK for testing, AAB for Play Store. Run npx expo install --fix whenever you upgrade Expo SDK to resolve dependency conflicts.

Section 12

Project Architecture

Layered Architecture

Ductivity follows a clean layered pattern that separates concerns:

Architecture Overview 
Screens (UI Layer)
  |-- TaskScreen, DashboardScreen, SettingsScreen
  |-- Consume services, render components
  |
Components (Reusable UI)
  |-- Toast, ActivityChart, SkeletonLoader
  |-- Pure presentational, receive data via props
  |
Context (Global State)
  |-- UserContext (username), NotificationContext (trigger counter)
  |
Services (Data Layer)
  |-- activityService (Firestore CRUD)
  |-- notificationService (push notifications)
  |-- settingsService (user preferences)
  |-- queryCache (in-memory TTL cache)
  |
Config & Types (Shared Foundation)
  |-- constants.ts (magic numbers)
  |-- types/index.ts (interfaces, enums)
  |-- firebase.ts (database init)

Service Layer Pattern

Screens never import Firestore directly. Instead, they call service functions that encapsulate all database logic. This makes it easy to swap backends, add caching, or change data transforms without touching UI code.

Pattern: Screen calls Service 
// In TaskScreen (UI layer):
const docId = await logActivity(activity.name, activity.category);

// In activityService (data layer):
export async function logActivity(...) {
  clearCache();
  const docRef = await addDoc(...);
  return docRef.id;
}
